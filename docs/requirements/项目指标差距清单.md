# 项目指标差距清单

## 当前结论

- 当前方案已经完成 RK3588 板端离线 TTS 跑通，能够在板端生成音频并自动回传日志与音频到 PC。
- 当前方案仍属于 PaddleSpeech TTSArmLinux + Paddle Lite CPU 基线方案，不是 RKNN NPU 加速方案。
- 按现有实测结果，暂时不能认定满足项目指标，主要差距集中在实时性、NPU 化、领域词发音评测、内存与长期稳定性验证。

## 1. 实时性指标差距

指标要求：

- 基于 RK3588 平台 NPU 硬件加速。
- 端侧离线 TTS 单句合成延迟 ≤ 200 ms。

当前状态：

- 当前板端实测日志显示单句 Inference time 为 718.381 ms。
- 当前运行链路使用的是 Paddle Lite CPU 推理，不是 RKNN NPU 推理。
- 现有基线句子可以合成成功，但距离 200 ms 仍有明显差距。

差距判断：

- 延迟约为目标值的 3.6 倍。
- “基于 NPU 硬件加速”这一前提条件当前未满足。

需要补齐的工程项：

1. 将声学模型与 vocoder 拆分为适合 RKNN Toolkit2 转换的静态图或固定 shape 推理链路。
2. 建立 RK3588 专用的 RKNN 推理工程，完成 CPU 前端 + NPU 推理 + CPU 后处理的混合流水线。
3. 完成 INT8 校准数据集准备与 RKNN 量化，分别验证声学模型和 vocoder 的精度损失。
4. 建立句长分桶与 padding 策略，避免动态 shape 导致的 NPU 推理失配。
5. 按 10 字、20 字、30 字、50 字短句统计 P50/P95 延迟，而不是只看单条样例。

建议验收口径：

- 在 20 到 50 条业务短句集合上，P50 与 P95 延迟都要小于 200 ms。
- 必须给出 RKNN 模型、板端 RKNN Runtime 日志和基准测试记录，证明使用了 NPU 路径。

## 2. 语音合成质量指标差距

指标要求：

- 中文合成准确率 ≥ 95%。
- 通信指令、专业词汇发音准确率 99%。

当前状态：

- 目前只验证了通用中文句子的可合成性，没有建立正式的准确率评测集。
- 当前前端仍是通用中文前端与通用词典，尚未针对通信指令、缩写、代号、频点、呼号等业务文本做专项优化。
- 带英文词的测试句子此前触发过前端断言，说明中英混排、专有名词、领域文本归一化能力仍不足。

差距判断：

- 当前没有客观评测数据支撑“中文准确率 ≥ 95%”。
- 当前没有专项词表和业务测试集，无法支撑“专业词汇 99% 准确率”。

需要补齐的工程项：

1. 建立业务领域文本清单，覆盖通信指令、呼号、数字串、频点、单位、专有名词、缩略语。
2. 为前端补充自定义词典、热词词典、拼音覆盖表和文本归一化规则。
3. 对中英混排、字母数字混排、专业缩写建立单独的前端转换策略，避免再次出现断言或错误发音。
4. 构建至少 500 条中文评测集和专项词汇评测集，给出逐句人工校验结果。
5. 给出 MOS、人工标注准确率或领域词命中率等正式质量报告。

建议验收口径：

- 中文通用句准确率与专项词准确率分开统计。
- 专项词汇必须附带词表、样例句、人工校验记录与错误归因。

## 3. 端侧轻量化与可靠性指标差距

指标要求：

- 模型经 INT8 量化后体积 ≤ 200 MB。
- 运行内存占用 ≤ 300 MB。
- 支持 7×24 小时不间断稳定运行。

当前状态：

- 当前本地运行包总体积约 160.5 MB。
- 当前模型目录体积约 99.57 MB。
- 仅从体积数值看，体积约束接近可满足。
- 但当前没有独立证据证明现用模型就是经过 RKNN INT8 量化后的最终交付模型。
- 当前也没有形成可信的峰值内存测量结果。
- 7×24 稳定性目前没有做长期压测与异常恢复验证。

差距判断：

- “体积小于 200 MB”从当前模型目录大小看问题不大，但“INT8 量化后”这一条件尚未形成可审计证据。
- “运行内存 ≤ 300 MB”当前没有正式测试报告，不能判定满足。
- “7×24 不间断稳定运行”当前完全没有 soak test 证据。

需要补齐的工程项：

1. 输出正式的交付物清单，标注模型格式、量化方式、量化校准集、模型大小与版本号。
2. 在板端实现可信的内存监测方法，记录单次推理峰值 RSS、稳态 RSS 和长时间运行内存漂移。
3. 建立长稳测试脚本，至少覆盖 24 小时、72 小时和 168 小时三个阶段。
4. 增加守护进程、异常退出自动拉起、日志滚动、磁盘空间保护和健康检查。
5. 验证高频调用、连续调用、空闲待机再唤起等场景下的稳定性与资源回收情况。

建议验收口径：

- 提供板端连续运行日志、内存曲线、崩溃统计和恢复记录。
- 至少一次连续 72 小时无人工干预运行通过，才能进入 7×24 验收阶段。

## 4. 优先级建议

1. 先完成 RKNN NPU 路线与 200 ms 延迟收敛，否则项目最核心指标无法成立。
2. 再补业务领域词典与前端归一化，否则质量指标无法量化验收。
3. 之后补内存测量、守护机制和长稳测试，形成可靠性交付闭环。
4. 最后整理正式验收材料，包括模型清单、性能报告、质量报告和稳定性报告。
