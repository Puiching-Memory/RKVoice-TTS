# 项目指标差距清单

## 说明

- 本清单基于 2026-04-13 最新综合报告更新，主要依据 artifacts/test-runs/rkvoice-report-20260413-010803/report.json、当前运行包 artifacts/runtime/sherpa_onnx_rk3588_runtime 和 docs/requirements/项目指标.md。
- 本文关注的是“距离指标验收还有哪些差距”，不再沿用早期“仓库只有 TTS 基线、尚无 ASR 模块”的旧判断。

## 当前结论

- 当前默认主线已经推进到 sherpa-onnx 离线 ASR + TTS 工程基线，仓库内已具备 ASR 的 CPU/ONNX 基线、ASR 的 RKNN/NPU 验证路径，以及 TTS 的 CPU/ONNX 基线运行包。
- 最新综合报告结论仍为 fail，当前指标汇总为 3 项通过、8 项未通过、4 项部分满足、8 项证据不足；单元测试 22/22 通过。
- 当前已经能够证明“离线运行包可组装、TTS/ASR 冒烟链路可跑通、ASR 路径已采到官方 RKNN profiler 证据、统一综合报告与测试计划已落地”。
- 当前仍不能认定满足 TTS 指标、ASR 指标和系统整体指标，主要差距已经从“有没有模块”转向“是否满足流式、TTS NPU、体积/内存、准确率、长稳和闭环验收”。

现有基线证据：

- 综合报告：artifacts/test-runs/rkvoice-report-20260413-010803/report.json
- 当前运行包目录：artifacts/runtime/sherpa_onnx_rk3588_runtime
- TTS 冒烟：单句 353 ms，音频时长 3.03 s，RTF 0.117，当前后端为 cpu-onnx。
- ASR RKNN 冒烟：单次 855 ms，RTF 0.153，NPU 峰值负载 52%。
- RKNN 官方 profiler：source = eval_perf，operator_count = 853，peak MacUsage = 92.28%，total memory = 511.36 MiB。
- 当前模型体积：TTS 模型 29.07 MiB 且未见 INT8 证据；ASR CPU 模型 226.131 MiB；ASR RKNN 模型 463.465 MiB；models/ 总量 905.876 MiB。
- 当前测试计划：共 8 个用例，覆盖 smoke、latency、domain、stability、acceptance 五类；其中 acceptance 用例已明确注明“当前 CPU 基线预计不会通过”。

## 1. 语音合成 TTS 指标差距

### 1.1 端侧合成延迟

指标要求：

- NPU 加速下，单句中文语音合成端到端延迟 ≤ 150 ms。

当前状态：

- 当前 TTS 后端仍为 cpu-onnx，不是 RKNN/NPU 路径。
- 最新综合报告中，单句 TTS 实测为 353 ms，较早期 PaddleSpeech CPU 基线已有改善，但仍高于 150 ms 目标。
- 当前测试计划里已经有 latency 和 acceptance 类用例，但 acceptance 说明明确写明“当前 CPU 基线预计不会通过”。

差距判断：

- 当前 TTS 延迟约为目标值的 2.35 倍。
- 更关键的问题不是“只差一点性能”，而是“指标要求的 NPU 前提当前根本没有满足”。
- 只继续优化现有 CPU 基线，不足以支撑最终验收口径。

需要补齐的工程项：

1. 明确 TTS 是否进入 RKNN 交付主线；如果答案是“是”，就不能再把 TTS 长期定位为 CPU 基线，只能把 CPU 路径保留为回归对照和兜底方案。
2. 建立 TTS 的 RK3588 专用推理路线，明确 CPU 前端、NPU 推理、CPU 后处理的边界和接口。
3. 为 10 字、20 字、30 字、50 字业务短句建立 P50/P95 延迟统计，而不是继续只看单条冒烟样例。
4. 输出可审计的 RKNN 模型、板端 RKNN Runtime 日志和性能记录，证明 TTS 实际命中 NPU。
5. 在 TTS 进入 NPU 路线前，现有 CPU 基线只应被视为“功能可跑通”，不应被描述为“接近验收”。

建议验收口径：

- 在 20 到 50 条业务短句集合上统计 P50 与 P95，且两者都应 ≤ 150 ms。
- 必须同时提供 RKNN 模型版本、量化说明、板端日志和统一报告截图，避免出现“看起来像 NPU、但无法审计”的情况。

### 1.2 合成准确率

指标要求：

- 普通话合成准确率 ≥ 98%。
- 通信指令、数字、字母发音准确无歧义。
- 支持粤语、四川话等主流方言合成，方言合成准确率 ≥ 95%。

当前状态：

- 当前测试计划已经包含 3 个 domain 类用例，说明“领域句集覆盖”这件事已经开始做，但当前报告还只是汇总计划覆盖和冒烟结果，并没有形成正式的准确率统计。
- 当前报告没有自动化音素、文本对齐或主客观评分结果。
- 当前运行包仍是中文普通话 TTS 基线，未见粤语、四川话等方言 TTS 交付证据。

差距判断：

- 当前可以证明“已经开始围绕领域文本组织测试计划”，但还不能证明“普通话准确率 ≥ 98%”。
- 当前没有专项词表、热词规则和人工校验记录，无法支撑“通信指令、数字、字母发音准确无歧义”。
- 当前没有方言模型和方言评测结果，无法支撑“方言准确率 ≥ 95%”。

需要补齐的工程项：

1. 把现有 domain 用例从“覆盖存在”升级为“有评分结果”的正式评测集，覆盖通信指令、呼号、数字串、频点、单位、缩写和字母数字混排。
2. 为前端补充自定义词典、热词词典、拼音覆盖表和文本归一化规则，重点解决数字、字母、代号和中英混排读法。
3. 建立人工校验模板和错误归因分类，保证准确率报告可以被复查。
4. 如果方言是硬指标，就必须补齐方言模型选型、部署路径和独立测试句集；如果方言暂不纳入本阶段交付，应该在项目范围文档中显式降级，而不是默认空缺。

建议验收口径：

- 普通话、专项词和方言三类指标必须分开统计。
- 所有准确率结论都应附带测试句集、人工校验记录、错误样例和错误归因。

### 1.3 模型与稳定性

指标要求：

- INT8 量化后模型体积 ≤ 100 MB。
- 运行内存占用 ≤ 300 MB。
- 支持 7×24 小时连续稳定运行。

当前状态：

- 当前 TTS 模型体积为 29.07 MiB，从纯体积看有空间，但报告明确显示 tts_model_is_int8 = false，当前没有 INT8 量化证据。
- 当前报告中的 tts_max_rss_mib 仍为 null，且没有可直接用于判断 TTS RSS 的 samples.csv 证据。
- 测试计划中已经存在 1 个 stability 类用例，但当前报告没有长时间连续运行数据，仍不能据此认定满足 7×24。

差距判断：

- “模型体积 ≤ 100 MB”在数值上不紧张，但“INT8 量化后”这个前提尚未满足，也没有可审计证据。
- “运行内存 ≤ 300 MB”当前仍属于未验证状态。
- “7×24 连续稳定运行”当前仍没有 soak test 证据，稳定性只能算计划已建立，不能算通过。

需要补齐的工程项：

1. 输出正式的模型清单，明确 TTS 模型格式、量化方式、校准集、版本号和文件大小。
2. 落地 TTS 进程级与系统级内存采样，至少输出单次峰值 RSS、稳态 RSS 和多轮调用后的内存漂移。
3. 将 stability 用例扩展为 24 小时、72 小时和 168 小时三阶段长稳脚本，配套日志滚动、磁盘保护、异常重启和健康检查。
4. 验证高频调用、连续调用、空闲待机再唤起等场景下的资源回收情况，避免“冒烟稳定、长跑失稳”。

建议验收口径：

- 提供板端连续运行日志、内存曲线、崩溃统计和恢复记录。
- 至少完成一次连续 72 小时无人工干预运行，才能进入 7×24 最终验收阶段。

### 1.4 运行模式

指标要求：

- 纯离线端侧运行，无需网络、无需云端接口。

当前状态：

- 当前 runtime 已标记 offline_ready = true。
- 当前 TTS 运行链路不依赖网络和云端接口，离线子链路已经成立。

差距判断：

- 对 TTS 子链路本身，这一项可以判定为已满足。
- 需要注意的是，“TTS 已离线”并不等于“整个系统离线闭环已完成”，系统级离线能力仍需在第 3 节单独判断。

需要补齐的工程项：

1. 在交付材料中补充离线依赖清单和断网 smoke 记录。
2. 后续系统级验收时，补做断网环境下的 ASR + TTS 闭环回归。

## 2. 语音识别 ASR 指标差距

### 2.1 端侧识别延迟

指标要求：

- 流式语音识别端到端延迟 ≤ 200 ms。

当前状态：

- 当前默认 ASR 模式为 offline，不是 streaming。
- 当前 ASR 已具备 RKNN/NPU 路径，且已采到官方 profiler 证据，这说明“ASR 命中 NPU”不再是空白项。
- 但最新综合报告里，ASR RKNN 单次耗时为 855 ms，明显高于 200 ms 指标。

差距判断：

- 当前 ASR 已从“未实现”推进到“已有离线 RKNN 基线”，这是重要进展。
- 但从验收口径看，当前同时卡在两个点上：一是“不是流式”，二是“延迟远高于 200 ms”。
- 因此 ASR 这项的主要问题不是“有没有 NPU”，而是“有没有流式链路和端到端时延收敛”。

需要补齐的工程项：

1. 明确 ASR 的最终交付形态是流式还是分段准实时；如果指标不变，就必须补齐真正的流式接口和时间戳。
2. 建立流式推理的音频缓存、分段、partial result 和 final result 机制，避免继续拿离线整段推理结果对标流式指标。
3. 对 1 s、3 s、5 s、10 s 输入片段统计 P50/P95 延迟，并区分首包延迟、最终结果延迟和整段完成时间。
4. 在统一报告里增加流式专用观测项，避免继续用离线 smoke 指标代替流式验收指标。

建议验收口径：

- 必须给出流式识别时间戳、chunk 大小、partial/final 产出时序和 P50/P95 延迟。
- 单看离线 RTF 或单次整段耗时，不能作为这项的最终验收依据。

### 2.2 识别准确率

指标要求：

- 普通话指令识别准确率 ≥ 95%（典型噪声环境）。
- 支持主流方言语音识别，方言识别准确率 ≥ 90%。

当前状态：

- 当前报告只汇总了冒烟转写样例，没有形成准确率统计。
- 当前报告没有低信噪比或短波链路样本，也没有方言 ASR 验证数据。

差距判断：

- 当前可以证明“ASR 能离线跑通，且已有普通话 smoke 转写样例”，但还不能证明“普通话指令准确率 ≥ 95%”。
- 当前没有方言和典型噪声环境数据，无法支撑方言准确率和抗噪指标。

需要补齐的工程项：

1. 建立普通话指令集、典型噪声环境测试集和主流方言测试集，并输出准确率统计方法。
2. 在报告中增加命令词准确率、CER/WER 或任务级正确率等可复核指标，而不是只展示个别 smoke 例句。
3. 引入错误样例清单，区分热词、数字、专有名词、口音、噪声和链路失真导致的问题。
4. 如果短波电台是实际业务场景，应尽早把短波失真样本并入 ASR 准确率评测，而不是等到系统联调再处理。

建议验收口径：

- 至少给出普通话指令、主流方言、典型噪声三类独立测试结果。
- 所有准确率统计都应附带样本来源、测试规模和错误样例。

### 2.3 模型与稳定性

指标要求：

- INT8 量化后模型体积 ≤ 100 MB。
- 支持流式识别。
- 支持 7×24 小时连续稳定运行。

当前状态：

- ASR CPU 模型已显示 asr_cpu_model_is_int8 = true，但体积为 226.131 MiB，已经超过 100 MB 指标。
- ASR RKNN 模型体积为 463.465 MiB，也远超 100 MB 指标。
- RKNN memory profile 显示 total memory = 511.36 MiB，仅 ASR 单路径就已接近或超过系统典型工况内存上限。
- 当前报告没有长时间连续 ASR 运行数据。

差距判断：

- ASR 已不再是“零基线”，但模型体积这一项已经明确不满足。
- 流式能力本身尚未落地，因此“支持流式识别”当前也不能成立。
- 稳定性仍缺少 24 h 以上的持续运行证据。

需要补齐的工程项：

1. 重新审视 ASR 模型选型、压缩和量化策略，避免后续系统联调时才发现体积和内存根本装不下。
2. 区分“CPU 兜底模型”和“最终交付模型”，避免把多个大型模型同时打进默认运行包，拉高总存储占用。
3. 尽快补齐流式接口和长稳脚本，否则这一节三个指标会长期同时挂起。
4. 为 ASR 单独输出长时间稳定性、内存漂移和异常恢复记录。

建议验收口径：

- 给出交付态 ASR 模型清单和最终文件大小，而不是只统计开发态全部模型目录。
- 稳定性应至少包含 24 h 和 72 h 两阶段记录，再进入 7×24 验收。

### 2.4 运行模式

指标要求：

- 纯离线端侧运行，无需网络、无需云端接口。

当前状态：

- 当前 ASR 运行包仅暴露离线识别入口，不依赖云端接口。

差距判断：

- 对当前 ASR 子链路而言，这一项可以判定为已满足。
- 但这不等于系统级闭环已具备完整断网运行验收证据。

需要补齐的工程项：

1. 在交付材料中补充断网环境下的 ASR 冒烟记录。
2. 将后续闭环系统的离线依赖关系单独列出，避免系统级验收时再追溯组件依赖。

## 3. 系统整体指标差距

指标要求：

- 全链路闭环延迟（语音→文字→语音部分）≤ 350 ms。
- 模型总存储占用 ≤ 200 MB。
- 典型工况内存占用 ≤ 500 MB。
- 支持 RK3588 NPU 硬件加速，CPU 占用可控。
- 适配短波电台音频特性。
- 可与低码率语音压缩模块无缝对接。
- 支持普通话与主流方言的混合语音交互与播报。
- 系统整体全程离线独立工作，无任何外部网络依赖。

当前状态：

- 当前仓库已经不再是单一 TTS 子链路，而是 ASR + TTS 双子链路工程基线。
- 但当前仍没有语音→文字→语音的统一闭环调度入口，也没有系统级闭环时间戳和状态机。
- 当前 models/ 总量为 905.876 MiB，已远超 200 MB 目标，超出约 705.876 MiB。
- 当前 ASR 路径已采到官方 RKNN profiler 证据，但 TTS 仍是 CPU-ONNX，系统层面只能算“部分命中 NPU”。
- 当前 ASR 单路径内存画像为 511.36 MiB，TTS 侧内存证据仍为空，因此系统典型工况内存基本不能判定为满足。
- 当前报告没有低信噪比或短波链路样本，也没有压缩链路对接结果和混合方言系统级测试结果。

差距判断：

- “全链路闭环 ≤ 350 ms”当前仍无法评估，因为闭环入口和系统级时间戳尚未建立。
- “模型总存储 ≤ 200 MB”当前已经明确不满足，而且差距很大，后续必须把模型预算当成 P0 约束。
- “典型工况内存 ≤ 500 MB”当前从 ASR 单链路内存画像看就已经非常紧张，系统总内存大概率不满足。
- “支持 RK3588 NPU 硬件加速，CPU 占用可控”当前只能判定为部分满足，因为 NPU 主要命中在 ASR，TTS 主线仍是 CPU。
- “短波适配、低码率压缩对接、混合方言交互”当前都还没有有效证据支撑。
- “系统整体全程离线独立工作”虽然组件级离线已成立，但系统级闭环尚未完成，因此仍需单独补系统级断网回归。

需要补齐的工程项：

1. 建立统一的闭环入口，串起音频输入、ASR、状态机或语义处理、TTS 输出，并补系统级时间戳采集。
2. 把模型预算和内存预算前置，先决定“最终交付态只保留哪些模型”，再继续推进系统联调，否则闭环做完也可能因体积超标无法交付。
3. 把 TTS 是否进入 RKNN 主线作为系统级 P0 决策；如果 TTS 继续长期停留在 CPU 基线，系统级 NPU 加速指标基本无法成立。
4. 建立系统级内存、CPU、NPU 联合画像，避免继续用单模块数据代替系统级判断。
5. 补齐短波音频样本、低码率压缩链路对接和普通话/方言混合交互场景。
6. 在断网环境下执行完整闭环回归，形成系统级离线独立运行证明。

建议验收口径：

- 闭环延迟、总存储、总内存、CPU/NPU 占用必须以系统级报告给出，不能分别引用 TTS 和 ASR 子模块结果来拼接结论。
- 短波、压缩链路和混合方言交互都应附带日志、音频样例和明确场景说明。

## 4. 优先级建议

1. 先修正项目口径：当前仓库已不是“只有 TTS 基线”，而是“已有 ASR + TTS 工程基线，但距离指标验收仍有明显差距”。所有后续计划都应基于这个事实展开。
2. 把 TTS RKNN/NPU 路线列为 P0 决策项。如果项目目标仍然严格对齐现有指标，TTS 长期停留在 CPU-ONNX 将直接阻塞最核心的 TTS 性能验收。
3. 把 ASR 流式化和模型预算列为 P0/P1。当前 ASR 已有离线 RKNN 基线，但“非流式 + 模型超大”会直接阻塞 ASR 和系统两类验收。
