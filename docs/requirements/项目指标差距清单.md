# 项目指标差距清单

## 当前结论

- 2026-04-12 已完成一次 RK3588 板端离线 TTS 端到端冒烟，能够完成运行包构建、上板部署、语音合成、日志回传和音频回传。
- 当前仓库只包含 TTS 交付基线，尚无 ASR 模块、流式识别链路或语音→文字→语音闭环实现。
- 当前板端路径仍属于 PaddleSpeech TTSArmLinux + Paddle Lite CPU 基线方案，不是 RKNN NPU 加速交付路径。
- 因此当前只能证明“TTS 纯离线板端可跑通”，不能认定满足新的 TTS 指标、ASR 指标和系统整体指标。

现有基线证据：

- 合成日志：artifacts/runtime/paddlespeech_tts_armlinux_runtime/output/smoke_test.log
- 合成音频：artifacts/runtime/paddlespeech_tts_armlinux_runtime/output/smoke_test.wav
- 当前运行包目录：artifacts/runtime/paddlespeech_tts_armlinux_runtime
- 最新一次板端 smoke test 指标：Inference time 783.059 ms，WAV duration 3100 ms，RTF 0.2526。
- 当前运行包总体积约 162.10 MB，其中 models/ 目录约 99.57 MB。

## 1. 语音合成 TTS 指标差距

### 1.1 端侧合成延迟

指标要求：

- NPU 加速下，单句中文语音合成端到端延迟 ≤ 150 ms。

当前状态：

- 当前板端实测 Inference time 为 783.059 ms。
- 当前运行链路使用的是 Paddle Lite CPU 推理，不是 RKNN NPU 推理。

差距判断：

- 当前延迟约为目标值的 5.2 倍。
- “NPU 加速”这一前提条件当前未满足。

需要补齐的工程项：

1. 将声学模型与 vocoder 拆分为适合 RKNN Toolkit2 转换的静态图或固定 shape 推理链路。
2. 建立 RK3588 专用的 RKNN 推理工程，完成 CPU 前端 + NPU 推理 + CPU 后处理的混合流水线。
3. 完成 INT8 校准数据集准备与 RKNN 量化，分别验证声学模型和 vocoder 的精度损失。
4. 建立句长分桶与 padding 策略，避免动态 shape 导致的 NPU 推理失配。
5. 按 10 字、20 字、30 字、50 字短句统计 P50/P95 延迟，而不是只看单条样例。

建议验收口径：

- 在 20 到 50 条业务短句集合上，P50 与 P95 延迟都要 ≤ 150 ms。
- 必须给出 RKNN 模型、板端 RKNN Runtime 日志和基准测试记录，证明使用了 NPU 路径。

### 1.2 合成准确率

指标要求：

- 普通话合成准确率 ≥ 98%。
- 通信指令、数字、字母发音准确无歧义。
- 支持粤语、四川话等主流方言合成，方言合成准确率 ≥ 95%。

当前状态：

- 目前只验证了普通话单条冒烟句子的可合成性，没有形成正式的准确率评测集。
- 当前前端仍是通用中文前端与通用词典，尚未针对通信指令、数字串、字母串、缩写、代号、频点、呼号等业务文本做专项优化。
- 当前运行包没有提供粤语、四川话等主流方言模型或单独的方言播报入口。

差距判断：

- 当前没有客观评测数据支撑“普通话准确率 ≥ 98%”。
- 当前没有专项词表和业务测试集，无法支撑“通信指令、数字、字母发音准确无歧义”。
- 当前没有方言模型和方言评测结果，无法支撑“方言准确率 ≥ 95%”。

需要补齐的工程项：

1. 建立业务领域文本清单，覆盖通信指令、呼号、数字串、频点、单位、专有名词、缩略语和字母数字混排场景。
2. 为前端补充自定义词典、热词词典、拼音覆盖表和文本归一化规则。
3. 对数字、字母、缩写和中英混排建立单独的前端转换策略，避免错误归一化或歧义读法。
4. 增补粤语、四川话等方言模型选型、部署方式和独立测试句集。
5. 构建正式评测集并给出人工校验结果、错误类型归因和质量报告。

建议验收口径：

- 普通话、专项词和方言三类指标必须分开统计。
- 专项词和方言项必须附带词表、样例句、人工校验记录与错误归因。

### 1.3 模型与稳定性

指标要求：

- INT8 量化后模型体积 ≤ 100 MB。
- 运行内存占用 ≤ 300 MB。
- 支持 7×24 小时连续稳定运行。

当前状态：

- 当前 models/ 目录体积约 99.57 MB，仅从大小数值看接近阈值。
- 但当前没有独立证据证明现用模型就是经过 RKNN INT8 量化后的最终交付模型。
- 当前没有形成可信的峰值内存测量结果。
- 7×24 稳定性目前没有做长期压测与异常恢复验证。

差距判断：

- “模型体积 ≤ 100 MB”从当前 models/ 目录大小看接近可满足，但“INT8 量化后”这一条件尚未形成可审计证据。
- “运行内存 ≤ 300 MB”当前没有正式测试报告，不能判定满足。
- “7×24 连续稳定运行”当前完全没有 soak test 证据。

需要补齐的工程项：

1. 输出正式的交付物清单，标注模型格式、量化方式、量化校准集、模型大小与版本号。
2. 在板端实现可信的内存监测方法，记录单次推理峰值 RSS、稳态 RSS 和长时间运行内存漂移。
3. 建立长稳测试脚本，至少覆盖 24 小时、72 小时和 168 小时三个阶段。
4. 增加守护进程、异常退出自动拉起、日志滚动、磁盘空间保护和健康检查。
5. 验证高频调用、连续调用、空闲待机再唤起等场景下的稳定性与资源回收情况。

建议验收口径：

- 提供板端连续运行日志、内存曲线、崩溃统计和恢复记录。
- 至少一次连续 72 小时无人工干预运行通过，才能进入 7×24 验收阶段。

### 1.4 运行模式

指标要求：

- 纯离线端侧运行，无需网络、无需云端接口。

当前状态：

- 当前 TTS 基线已经可以在板端离线运行，冒烟过程未依赖云端接口。

差距判断：

- 当前只能证明“TTS 子链路可离线运行”。
- 由于 ASR 与系统闭环尚未实现，不能把“全项目离线独立工作”判定为已满足。

需要补齐的工程项：

1. 输出离线依赖清单，明确最终交付系统在运行态不依赖外网服务。
2. 为后续 ASR 和闭环模块建立离线启动、自检和异常恢复流程。
3. 在验收阶段补做断网环境下的全链路独立运行测试。

## 2. 语音识别 ASR 指标差距

### 当前状态

- 当前仓库内没有 ASR 模块、流式识别 runtime、ASR 模型目录或 ASR 测试脚本。
- 当前也没有普通话、方言、噪声环境下的识别评测集与验收报告。

### 差距判断

- ASR 端侧识别延迟、识别准确率、模型与稳定性、运行模式四类指标当前均处于“未实现或未验证”状态。
- 在 ASR 基线尚未落地前，无法对 200 ms 延迟目标、95% 普通话准确率、90% 方言准确率或 7×24 稳定性做任何有效判定。

### 需要补齐的工程项

1. 明确 ASR 技术路线，选定纯离线、可流式、适配 RK3588 的识别方案。
2. 完成 ASR 模型部署、流式推理接口、音频缓存与分段策略。
3. 准备普通话指令集、主流方言数据集和典型噪声环境测试集。
4. 给出 ASR INT8 量化方案、模型体积记录和稳定性验证计划。
5. 建立 ASR 单独的板端延迟、准确率和长稳测试脚本。

### 建议验收口径

- 至少给出普通话指令、方言指令和噪声环境三类独立测试结果。
- 必须提供流式识别端到端时间戳、识别错误样例和模型版本记录。

## 3. 系统整体指标差距

指标要求：

- 全链路闭环延迟（语音→文字→语音部分）≤ 350 ms。
- 模型总存储占用 ≤ 200 MB。
- 典型工况内存占用 ≤ 500 MB。
- 支持 RK3588 NPU 硬件加速，CPU 占用可控。
- 适配短波电台音频特性。
- 可与低码率语音压缩模块无缝对接。
- 支持普通话与主流方言的混合语音交互与播报。
- 系统整体全程离线独立工作，无任何外部网络依赖。

当前状态：

- 当前仓库只有 TTS 子链路，尚未形成语音→文字→语音闭环。
- 当前运行包仅 TTS 一项就约 162.10 MB，如果后续加入 ASR，距离“模型总存储 ≤ 200 MB”的预算空间只剩约 37.90 MB。
- 当前没有全系统内存测量、CPU/NPU 占用记录，也没有低码率语音压缩模块对接结果。
- 当前没有普通话与主流方言混合交互测试。

差距判断：

- “全链路闭环 ≤ 350 ms”当前无法评估，因为闭环尚未实现。
- “模型总存储 ≤ 200 MB”在当前 TTS 基线下已经较紧张，后续 ASR 选型必须严格受控。
- “典型工况内存 ≤ 500 MB”“CPU 占用可控”“低码率压缩无缝对接”“混合语音交互”当前均无证据支撑。

需要补齐的工程项：

1. 明确闭环架构，打通音频输入、ASR、语义处理、TTS 输出和状态机调度。
2. 为 TTS 与 ASR 制定统一的存储预算、内存预算和 CPU/NPU 占用预算。
3. 建立闭环时间戳采集方案，统计 P50/P95 全链路时延。
4. 完成与低码率语音压缩模块的接口适配、音质回归和延迟测试。
5. 增补普通话与主流方言混合交互场景的系统级测试方案。
6. 在断网环境下执行完整闭环回归，证明系统整体可独立运行。

建议验收口径：

- 闭环延迟、总存储、总内存、CPU/NPU 占用必须以系统级报告给出，而不是分别引用 TTS 和 ASR 子模块数据。
- 低码率压缩与混合语音交互必须附带场景说明、日志、音频样例和回归结论。

## 4. 优先级建议

1. 先明确项目范围是否已经从“单一 TTS 交付”扩展为“TTS + ASR + 闭环系统”交付；如果是，必须尽快补齐 ASR 基线，否则系统级指标无法进入实现阶段。
2. 并行推进 TTS 的 RKNN NPU 路线与 150 ms 延迟收敛，否则最核心的 TTS 性能指标无法成立。
3. 之后补普通话、专项词、数字字母和方言评测集，形成可量化的质量验收材料。
4. 再补总存储、总内存、守护机制和长稳测试，形成系统级可靠性交付闭环。
5. 最后整理正式验收材料，包括模型清单、性能报告、质量报告、闭环时延报告和稳定性报告。
