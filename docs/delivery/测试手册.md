# 测试手册

## 1. 目标

本手册用于说明当前仓库可直接执行的测试方式、证据位置，以及这些测试与项目指标的对应关系。

## 2. 当前范围

- 当前仓库直接覆盖的测试范围：TTS 单元测试、板端部署、单句板端冒烟、板端 profile 工具。
- 当前仓库尚未提供：独立批量 TTS 验收 CLI、ASR 流式识别测试、语音→文字→语音闭环测试。

## 3. 当前测试入口

单元测试入口：

```powershell
uv run python -m tests
```

板端端到端冒烟入口：

```powershell
uv run python -m scripts.delivery.paddlespeech_tts_armlinux all
```

如需拆开执行，也可以按 download / build / upload 三步运行：

```powershell
uv run python -m scripts.delivery.paddlespeech_tts_armlinux download
uv run python -m scripts.delivery.paddlespeech_tts_armlinux build
uv run python -m scripts.delivery.paddlespeech_tts_armlinux upload
```

板端上传后的运行包自带 profile 工具：

- tools/board_profile_capabilities.sh：检查 perf、strace、rknpu/load 等能力是否可用。
- tools/profile_tts_inference.sh：包装 run_tts.sh，额外采集进程 RSS、线程数、utime/stime 与 rknpu load 采样。

手工查看板端 profile 能力：

```bash
cd /root/tts/paddlespeech_tts_armlinux_runtime
./tools/board_profile_capabilities.sh
```

手工采集单句 profile：

```bash
cd /root/tts/paddlespeech_tts_armlinux_runtime
./tools/profile_tts_inference.sh \
    --sentence "短波电台指令测试。" \
    --output_wav ./output/profile.wav \
    --log ./output/profile.log \
    --samples-csv ./output/profile-samples.csv
```

## 4. 当前可直接产出的证据

- 单元测试通过结果。
- 板端冒烟日志：artifacts/runtime/paddlespeech_tts_armlinux_runtime/output/smoke_test.log。
- 板端冒烟音频：artifacts/runtime/paddlespeech_tts_armlinux_runtime/output/smoke_test.wav。
- 板端 profile 采样结果：运行包 output/ 目录下的 profile.log、profile.wav、profile-samples.csv。

## 5. 与项目指标的对应关系

当前测试能够直接支撑的内容：

- TTS 纯离线端侧运行可跑通。
- TTS 运行包可构建、可部署、可在板端生成音频并回传。
- TTS 当前 CPU 基线的单句推理延迟、RTF 和板端 profile 数据可采集。

当前测试不能直接替代的内容：

- TTS 最终 NPU 验收指标：单句中文语音合成端到端延迟 ≤ 150 ms。
- ASR 指标：流式识别 ≤ 200 ms、普通话与方言识别准确率、ASR 7×24 稳定性。
- 系统整体指标：语音→文字→语音闭环 ≤ 350 ms、总模型存储 ≤ 200 MB、典型工况内存 ≤ 500 MB、低码率语音压缩集成与混合方言交互。

## 6. 建议落地方式

按阶段拆分测试：

1. 日常联调保留 uv run python -m tests 和板端 smoke test，确保当前 TTS 基线不退化。
2. 进入 NPU 优化阶段后，基于 board_profile_capabilities.sh 和 profile_tts_inference.sh 采集 TTS 延迟、RSS 与 NPU load 数据，对标 150 ms 目标。
3. ASR 落地后新增独立的流式识别测试集、噪声环境测试集和方言测试集。
4. 闭环集成后新增系统级时延、总存储、总内存和低码率语音压缩联调测试。

按验收指标拆分阈值：

1. 基线阈值用于保证当前 CPU 方案不退化。
2. TTS 验收阈值用于对标最终 NPU 指标，例如 150 ms 目标。
3. ASR 和闭环系统必须独立维护自己的验收阈值，不要混入当前 TTS 基线口径。
