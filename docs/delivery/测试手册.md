# 测试手册

## 1. 目标

本手册用于说明当前仓库可直接执行的测试方式、证据位置，以及这些测试与项目指标的对应关系。

## 2. 当前范围

- 当前仓库直接覆盖的测试范围：Python 单元测试、sherpa-onnx RK3588 运行包组装与上板、板端 smoke test、板端 profile 工具、综合 HTML/JSON 报告生成。
- 当前仓库尚未提供：ASR 流式识别测试集、语音→文字→语音闭环自动化测试、带标注的 TTS/ASR 准确率评测集。

## 3. 当前测试入口

单元测试入口：

```powershell
uv run python -m tests
```

板端端到端冒烟入口：

```powershell
uv run python -m scripts.delivery.sherpa_onnx_rk3588 all
```

如需拆开执行，也可以按 download / build / upload 三步运行：

```powershell
uv run python -m scripts.delivery.sherpa_onnx_rk3588 download
uv run python -m scripts.delivery.sherpa_onnx_rk3588 build
uv run python -m scripts.delivery.sherpa_onnx_rk3588 upload
```

综合测试报告入口：

```powershell
uv run python -m scripts.testing.rkvoice_report
```

Toolkit2 Docker profiling 入口：

```powershell
uv run python scripts/testing/rknn_toolkit2_profile_in_docker.py --adb-connect 192.168.1.10:5555
```

如需先修复板端 adbd 5037 监听并重启 rknn_server：

```powershell
uv run python scripts/board/prepare_rknn_debug_bridge.py
uv run python scripts/testing/rknn_toolkit2_profile_in_docker.py --prepare-board-debug-bridge
```

报告默认会：

- 重新执行 unittest discovery。
- 扫描 artifacts/runtime/sherpa_onnx_rk3588_runtime/output/ 下的 smoke / profile 证据。
- 读取 config/local/tts_test_plan.json；若不存在，则回退到 config/examples/tts_test_plan.example.json。
- 读取 docs/requirements/项目指标.md 并生成逐条指标矩阵。
- 输出 artifacts/test-runs/rkvoice-report-时间戳/index.html 与 report.json。

板端上传后的运行包自带 profile 工具：

- tools/board_profile_capabilities.sh：检查 Python / librknnrt / rknn_toolkit2 / rknnlite2、RKNN_LOG_LEVEL=4 与 rknpu/load 等 profiling 入口是否可用。
- tools/profile_asr_inference.sh：包装 RKNN ASR 冒烟，自动采集 RKNN_LOG_LEVEL=4 运行时层日志与 rknpu/load 采样。
- tools/profile_tts_inference.sh：包装 run_tts.sh，额外采集进程 RSS、线程数、utime/stime 与 rknpu load 采样。

Host 侧 Docker profiling 工具：

- scripts/board/prepare_rknn_debug_bridge.py：通过 SSH 检查板端 adbd 是否监听 5037；若缺失，则按 RKNN Toolkit2 官方文档下载并替换 linux-aarch64 adbd，再重启 rknn_server。
- scripts/testing/rknn_toolkit2_profile_in_docker.py：构建 Linux x86_64 + Python 3.12 + RKNN Toolkit2 2.3.2 镜像，在容器内执行 host-side eval_perf / eval_memory，并把结果写回运行包 output/。
- docker/toolkit2-profile/Dockerfile：Toolkit2 profiling 镜像定义，内置 adb 与 rknn-toolkit2。

使用前提：

- Docker Desktop 或等价 Linux Docker daemon 可用。
- RK3588 板卡需提供 adb over TCP 或其他容器可达的 adb 连接方式。
- 容器默认不会替你开启 adb tcpip；如果板端只有 SSH 可达，这条 host-side Toolkit2 流程仍然跑不起来。
- 当前 A588 Ubuntu 固件若只监听 5555、没有 5037，可先执行 scripts/board/prepare_rknn_debug_bridge.py；该脚本会保留 adbd 备份并把 rknn_server 重新拉起。

手工查看板端 profile 能力：

```bash
cd /root/rkvoice/sherpa_onnx_rk3588_runtime
./tools/board_profile_capabilities.sh
```

手工采集单句 profile：

```bash
cd /root/rkvoice/sherpa_onnx_rk3588_runtime
./tools/profile_tts_inference.sh \
    --sentence "短波电台指令测试。" \
    --output_wav ./output/profile.wav \
    --log ./output/profile.log \
    --samples-csv ./output/profile-samples.csv
```

## 4. 当前可直接产出的证据

- 单元测试通过结果。
- 综合测试报告：artifacts/test-runs/rkvoice-report-时间戳/index.html 与 report.json。
- 板端冒烟日志：artifacts/runtime/sherpa_onnx_rk3588_runtime/output/smoke_test_summary.log。
- 板端冒烟音频：artifacts/runtime/sherpa_onnx_rk3588_runtime/output/smoke_test_tts.wav。
- 板端能力快照：artifacts/runtime/sherpa_onnx_rk3588_runtime/output/board_profile_capabilities.txt。
- RKNN 官方 profiler 原始产物：运行包 output/ 下的 rknn_eval_perf.txt、rknn_query_perf_detail.txt、rknn_perf_run.json、rknn_memory_profile.txt（若已生成）。
- Toolkit2 Docker profiling 额外会生成：rknn_sdk_version.json、rknn_adb_devices.txt、rknn_toolkit2_profile_manifest.json。
- RKNN 运行时层日志：artifacts/runtime/sherpa_onnx_rk3588_runtime/output/rknn_runtime.log。
- RKNN NPU 时序采样：artifacts/runtime/sherpa_onnx_rk3588_runtime/output/rknpu_load.log（同时保留兼容文件名 rknn_profile.log）。
- TTS profile 采样结果：运行包 output/ 目录下的 profile.log、profile.wav、profile-samples.csv。

## 5. 与项目指标的对应关系

当前测试能够直接支撑的内容：

- sherpa-onnx ASR + TTS 运行包可构建、可部署、可在板端完成 smoke test。
- 综合报告可把 unittest、RKNN 官方 profiler 层级明细、板端日志、音频预览、NPU 时序热力图和项目指标矩阵集中展示。
- TTS 当前 CPU 基线的单句推理延迟、RTF 和板端 profile 数据可采集。
- ASR RKNN 路径可通过 eval_perf / PERF_DETAIL / PERF_RUN / MEM_SIZE 原始文件、RKNN_LOG_LEVEL=4 日志和 rknpu/load 采样三层证据链体现。
- 若 adb over TCP 可用，Docker 化 Toolkit2 流程可以直接为现有报告补齐 eval_perf / eval_memory / perf_run 文件，而无需在 Windows 主机本地安装 Toolkit2。

当前测试不能直接替代的内容：

- TTS 最终 NPU 验收指标：单句中文语音合成端到端延迟 ≤ 150 ms。
- ASR 指标：流式识别 ≤ 200 ms、普通话与方言识别准确率、ASR 7×24 稳定性。
- 系统整体指标：语音→文字→语音闭环 ≤ 350 ms、总模型存储 ≤ 200 MB、典型工况内存 ≤ 500 MB、低码率语音压缩集成与混合方言交互。

## 6. 建议落地方式

按阶段拆分测试：

1. 日常联调保留 uv run python -m tests 和 uv run python -m scripts.testing.rkvoice_report，确保当前基线不退化。
2. 板端 smoke test 后再运行综合报告，让 HTML Dashboard 始终反映最新 runtime output 证据。
3. 若板端或外部工具已生成 eval_perf / PERF_DETAIL / PERF_RUN / MEM_SIZE 结果，只需把文件放到 output/，综合报告会自动吸收并渲染层级 profiler 面板。
4. 进入 NPU 优化阶段后，基于 profile_tts_inference.sh 补齐 TTS RSS / NPU 采样，并将其接入报告中的时序热力图。
5. ASR 流式链路、闭环链路和低码率压缩链路成熟后，再分别补充独立测试集和自动判定规则。

按验收指标拆分阈值：

1. 基线阈值用于保证当前 CPU 方案不退化。
2. TTS 验收阈值用于对标最终 NPU 指标，例如 150 ms 目标。
3. ASR 和闭环系统必须独立维护自己的验收阈值，不要混入当前 TTS 基线口径。

## 7. 报告解读注意事项

- index.html 中的 RKNN 官方 profiling 面板优先展示层级 Time(us) / NPU Cycles / MacUsage；热力图只是采样视图，不是 perf 调用栈火焰图。
- 若缺少 profile-samples.csv，报告仍会生成，但 TTS RSS 和 TTS heatmap 会显示为证据不足。
- 若只跑了 unittest、没有最新板端 output，报告中的 requirements matrix 会偏向 unknown 或 partial，而不是误判为 pass。
